# v12 — 방향키 중앙 AUTO / RUN 삭제

## 변경 요약

- **AUTO 버튼을 방향키 중앙으로 이동.** 기존에는 방향키 옆에 별도로 떠 있었으나, 이제 `#dpad`의 중앙 셀 자체가 AUTO 버튼입니다. 한 손 엄지 조작에 맞춘 배치입니다. 경로 이동 중에는 금색(`.active`)으로 바뀌고, 이동 중 다시 탭하면 즉시 취소됩니다.
- **RUN 버튼 완전 삭제.** 버튼, CSS, Shift 가속 로직 모두 제거했습니다. 대신 기본 이동속도를 15 → 17 (약 13% 증가)로 올려 RUN 없이도 답답하지 않게 조정했습니다. 기존에 저장된 게임(구버전 속도 15로 저장됨)도 불러올 때 자동으로 17로 올라갑니다.

## 수정된 버그

| 항목 | 내용 |

## 알려진 미해결 이슈

**도로 모서리 미세 떨림은 이번 버전에서 고치지 못했습니다.** 처음에는 노드 스냅 임계값을 넓혀서(`.006/.994` → `.012/.988`) 해결을 시도했는데, 자체 제작한 이동 시뮬레이터로 실제 돌려본 결과 **오히려 이동이 완전히 멈추는 심각한 회귀 버그**였습니다 (긴 도로 구간에서 프레임당 이동량이 넓어진 임계값보다 작아서, 매 프레임 시작 노드로 되돌아가며 전진하지 못함). 그래서 원래 임계값으로 되돌렸고, 모서리 떨림 자체는 다른 방식(예: 방향 선택에 짧은 재평가 쿨다운을 두는 방식)으로 별도 후속 작업이 필요합니다.

이 리포지토리에는 이전 버전들의 `FINAL_*_QA.json` 파일이 다수 있는데, 재현 가능한 근거 없이 "통과"라고만 적혀 있는 것들이 많았습니다. 이번 `FINAL_V12_DPAD_CENTER_AUTO_QA.json`은 실제로 실행한 명령과 그 결과만 기록했고, 브라우저/모바일 기기에서 직접 확인하지 못한 항목(포인터 캡처 실제 동작, safe-area 실제 렌더링, 화면 회전 타이밍)은 "검증 안 됨"으로 명시했습니다. 배포 전 실제 iPhone/Android 기기에서 가로 모드로 한 번 확인해보시길 권장합니다.

# v13 — AUTO 버튼 크기 버그 + 이동 재검증

## 스크린샷에서 발견된 문제

v12에서 AUTO 버튼에 "AUTO" 텍스트가 들어가면서, 명시적 크기 지정이 없어 CSS Grid가 콘텐츠 크기 기준으로 셀을 확장시켜 버튼이 타원형으로 커지고 아래쪽 방향키를 침범했습니다. "세로 이동이 걸린다"는 증상은 이 시각적 침범 때문일 가능성이 높다고 판단했습니다.

## 수정 내용

CSS Grid를 완전히 버리고, 5개 버튼(상/하/좌/우/AUTO) 모두 **고정 px 절대좌표**로 재배치했습니다. 각 버튼은 정확히 46.667px 정사각형이고 인접 버튼과 6px 여백을 둡니다. **실제 헤드리스 브라우저(Playwright+Chromium)로 렌더링해서 스크린샷 및 좌표 수치로 직접 검증**했고, 5개 버튼이 겹침 없이 정확한 십자 배열로 나오는 것을 확인했습니다.

## 정직하게 밝히는 미해결 사항

검증 과정에서 예상치 못한 현상을 발견했습니다: **실제 합성 마우스/포인터 이벤트(Chromium DevTools Protocol)를 왼쪽 방향키(◀) 정중앙 좌표에 쏘면, 좌표상 겹치지 않는데도 이벤트가 AUTO 버튼으로 전달되는 경우가 반복적으로 관찰**됐습니다. `getBoundingClientRect()` 수치와 시각적 스크린샷 모두 겹침이 없다고 나왔고, `document.elementsFromPoint()`(다중 히트테스트 API)는 정확하게 moveLeft를 최상단으로 반환했지만, 단일 `elementFromPoint()`와 CDP 합성 클릭만 계속 틀린 답을 줬습니다.

CSS를 Grid → 변수 기반 절대좌표 → 완전 하드코딩 px까지 세 가지 방식으로 바꿔가며 재현을 시도했고, z-index·box-shadow·border·text-shadow·touch-action·transform·중복 ID 등 여러 가설을 하나씩 제거하며 테스트했지만 원인을 확정하지 못했습니다. game.js 없이 순수 정적 HTML+CSS만으로도 재현되어 JS 로직 문제는 아님을 확인했지만, dpad 마크업만 따로 떼어 인라인 스타일로 테스트하면 재현되지 않고 실제 style.css 전체를 로드해야만 재현되는 등, 스타일시트의 다른 부분과의 상호작용이 있는 것으로 보이는데 정확히 어떤 규칙 때문인지는 시간 관계상 끝까지 못 찾았습니다.

**이것이 실제 사용자에게 영향을 주는 버그인지는 확인되지 않았습니다.** 헤드리스 Chromium의 `elementFromPoint`/CDP 합성 입력은 실제 기기의 OS 레벨 터치 입력과 다른 코드 경로를 타는 경우가 있어, 이 환경에서만 나타나는 현상일 가능성도 있습니다. 원인 규명에 시간을 계속 쓰는 것이 무책임하다고 판단해 조사를 중단하고, 원인과 무관하게 도움이 되는 방어적 조치(정확한 크기, 버튼 간 6px 여백, 불필요한 z-index/의사요소 제거)를 적용한 상태로 전달합니다.

**실제 휴대폰에서 AUTO 버튼 근처 왼쪽 방향키를 눌러봐서 정상 작동하는지 꼭 확인해주세요.** 만약 문제가 재현되면 기기/브라우저 정보와 함께 알려주시면, 이번에 못 찾은 실제 터치 입력 기반 재현이 가능해집니다.
